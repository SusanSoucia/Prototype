## 一、现状与问题

### 1.1 前端问题

| # | 问题 | 说明 |
|---|------|------|
| 1 | **缺少角色路由** | 当前所有路由（聊天、知识库、组织标签）对 ADMIN 和 USER 均可见，未做角色区分 |
| 2 | **缺少管理员专属页面** | 无用户监控页（全员聊天记录、Token 用量）、无 API 剩余用量页 |
| 3 | **缺少用户的聊天记录页** | 用户只能看当前会话，没有独立的聊天历史汇总页 |
| 4 | **知识库页面未区分部门** | 当前从 `/documents/accessible` 取所有文档，未按部门隔离 |

### 1.2 后端问题

| # | 问题 | 说明 |
|---|------|------|
| 1 | **知识库混合存储** | 所有部门文档存在同一个 ES 索引 `knowledge_base` 中，仅靠查询时 `orgTag` 字段过滤做逻辑隔离 |
| 2 | **Controller 层大面积缺失** | UserService、OrgTagCacheService 等业务逻辑已完整实现，但缺少 REST Controller 暴露 HTTP 端点（详见 `backend-功能实现分析.md`） |
| 3 | **Kafka Consumer 未实现** | 文件异步处理链路（解析→切块→向量化→ES索引）中断，`consumer/` 包为空 |
| 4 | **路径前缀不匹配** | 后端用 `/api/v1/` 前缀，前端请求不带前缀，需在 Nginx/Vite 代理层统一 |
| 5 | **缺少库管角色** | 当前只有 USER/ADMIN 两种角色，缺少中间层的知识库管理员 |

## 三、权限与标签架构（核心设计）

### 3.1 标签树：ADMIN 树和 USER 树完全分离

```
ADMIN 树（parentTag=null）                 USER 树（parentTag=null）
├── 研发部库管 (parentTag=ADMIN)           ├── 研发-访问 (parentTag=USER)
├── 市场部库管 (parentTag=ADMIN)           │   ├── 研发-算法组 (parentTag=研发-访问)
└── 人事部库管 (parentTag=ADMIN)           │   └── 研发-工程组 (parentTag=研发-访问)
                                           ├── 市场-访问 (parentTag=USER)
                                           └── 跨部门项目组 (parentTag=USER)
```

**核心规则：**
- 一个人不会同时挂在两棵树下。标签追溯到根，要么是 ADMIN 树，要么是 USER 树
- 直接持有 `ADMIN` 根标签 = 超级管理员，仅 1 人
- 持有 `ADMIN` 子标签 = 库管
- 持有 `USER` 子标签 = 普通用户

### 3.2 角色判定（纯标签驱动，不依赖 User.Role 字段）

| 判定 | 逻辑 |
|------|------|
| 超级管理员 | 用户标签集合中包含 `ADMIN` |
| 库管 | 用户标签向上追溯到 parentTag=ADMIN 的那一层 → 库管标签的 tagId 映射到 ES 索引名 `knowledge_base_{tagId}` |
| 普通用户 | 用户标签向上追溯到 parentTag=USER 的那一层 → 通过映射表查找可访问的知识库 |

### 3.3 知识库访问权限：多对多映射

新增表 `kb_tag_access`：

```sql
CREATE TABLE kb_tag_access (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    kb_name     VARCHAR(100) NOT NULL,    -- ES 索引名，如 knowledge_base_研发部
    tag_id      VARCHAR(255) NOT NULL,    -- USER 树下的标签（子标签自动继承）
    created_by  VARCHAR(64),
    created_at  DATETIME,
    UNIQUE KEY uk_kb_tag (kb_name, tag_id)
);
```

**权限继承规则：**
- tag A 映射到库 K → tag A 及其所有子标签（USER 树下）均可检索库 K
- 用户标签的访问库 = **所有标签映射库的并集**
- 继承计算在查询时做（SQL 查闭包），不写入冗余行

### 3.4 库管职责

| 操作 | 说明 |
|------|------|
| 管理库内文档 | 上传/删除/向量化 — 写入 `knowledge_base_{库管标签tagId}` |
| 授权 USER 标签 | 在 `kb_tag_access` 中增删 `(自己的库, USER树标签)` 映射 |
| 不能做什么 | 不能动标签树结构（无权创建/删除 ADMIN 或 USER 树下的标签），不能管理用户（无权分配标签给用户） |

### 3.5 完整权限矩阵

| 能力 | 超级管理员 | 库管 | 普通用户 |
|------|:--:|:--:|:--:|
| 聊天（RAG） | ❌ | ✅ | ✅ |
| 查看自己的聊天记录 | ❌ | ✅ | ✅ |
| 查看全员聊天记录 | ✅ | ❌ | ❌ |
| 查看全员 Token 用量 | ✅ | ❌ | ❌ |
| 创建/删除向量库（ES 索引） | ✅ | ❌ | ❌ |
| 管理组织标签树（创建/删除/编辑标签） | ✅ | ❌ | ❌ |
| 分配标签给用户 | ✅ | ❌ | ❌ |
| 上传文档到自己的库 | ❌ | ✅ | ❌ |
| 删除自己库的文档 | ❌ | ✅ | ❌ |
| 授权 USER 标签访问自己的库 | ❌ | ✅ | ❌ |
| 检索知识库（按授权） | ❌ | ✅（仅自己的库） | ✅（仅授权的库） |
| 查看 API 剩余用量 | ✅ | ❌ | ❌ |
| 管理 AI 模型提供商配置 | ✅ | ❌ | ❌ |

### 3.6 向量库生命周期（超管操作）

创建向量库 = 三件事原子执行：

```
超管创建向量库 "研发部"
  → (1) ES 创建索引 knowledge_base_研发部（复用 EsIndexInitializer 的 mapping）
  → (2) ADMIN 树下创建子标签 "研发部库管"（parentTag=ADMIN）
  → (3) 将 ADMIN 子标签分配给对应库管用户
```

删除向量库：

```
超管删除向量库 "研发部"
  → (1) ES 删除索引 knowledge_base_研发部（数据不可恢复）
  → (2) 删除 kb_tag_access 中 kb_name 匹配的所有映射行
  → (3) 删除 ADMIN 树下对应的子标签 "研发部库管"
  → (4) 将该标签从所有用户身上移除
```

这样 ADMIN 树下的子标签和 ES 索引一一对应，不会出现孤儿标签或孤索引。

### 3.7 权限判定流程

```
请求到达 → JWT 解析 username
  → User.orgTags 查用户持有的标签列表
  → 对每个标签向上追溯，判定根标签是 ADMIN 还是 USER
  → ADMIN 树：库管 → 可写 knowledge_base_{库管标签tagId}
  → USER 树：查 kb_tag_access → 取所有 kb_name 的并集 → 可检索
  → 标签追溯 + 继承全部在内存计算，不查多次数据库
```

## 四、ES 索引级隔离方案

### 4.1 索引命名规则

```
knowledge_base_{tagId}

示例:
  knowledge_base_ADMIN           ← 超级管理员的知识库（可选，存公共文档）
  knowledge_base_研发部库管        ← 研发部的知识库
  knowledge_base_市场部库管        ← 市场部的知识库
  knowledge_base_人事部库管        ← 人事部的知识库
```

### 4.2 写入时路由

```java
// 伪代码 — 上传文档时
String adminParentTag = findAdminParentTag(user.getOrgTags());
String indexName = "knowledge_base_" + adminParentTag;
elasticsearchService.bulkIndex(documents, indexName);
```

### 4.3 检索时多索引查询

```java
// 伪代码 — 普通用户检索时
List<String> accessibleKbs = getAccessibleKbs(user);  // 从 kb_tag_access + 继承计算
List<String> indexNames = accessibleKbs.stream()
    .map(kb -> "knowledge_base_" + kb)
    .toList();
List<SearchResult> results = hybridSearchService.searchAcrossIndices(query, indexNames, topK);
```

若只有一个索引可直接搜单索引，多个索引则分别搜再按分数合并排序。

### 4.4 需要改动的代码位置

| 文件 | 改动 |
|------|------|
| `ElasticsearchService.java` | `bulkIndex()` 接受 `indexName` 参数，不再硬编码 `knowledge_base` |
| `ElasticsearchService.java` | `deleteByFileMd5()` 接受 `indexName` 参数 |
| `ElasticsearchService.java` | `countByFileMd5()` 接受 `indexName` 参数 |
| `HybridSearchService.java` | `searchWithPermission()` 改为接受索引名列表 |
| `EsIndexInitializer.java` | 支持动态创建索引（按 `tagId` 生成 mapping） |
| `UserService.java` | 新增 `findAdminParentTag()` 方法 — 标签向上追溯 |
| 新增 | `KbTagAccessService.java` — 知识库-标签映射管理 |
| 新增（数据库） | `kb_tag_access` 表 |
| 新增 | `KbTagAccessRepository.java` |
| 新增 | `KbTagAccessController.java` — REST 端点供库管管理 |

### 4.5 User.Role 字段处理

保留 `User.Role` 枚举但不作为鉴权依据，新增 `LIBRARY_ADMIN` 值仅用于：
- 前端路由判定（哪些页面可见）
- 快速筛选用户列表
- 权限由标签层级计算，`Role` 仅为缓存/辅助字段，写入时与标签同步

```java
public enum Role {
    USER,           // 普通用户（USER 树下）
    LIBRARY_ADMIN,  // 库管（ADMIN 子标签持有者）
    ADMIN           // 超级管理员（ADMIN 标签持有者）
}
```

### 6.4 Kafka Consumer 实现（优先级：中）

文件上传异步处理链路：

```
上传分片 → 合并完成 → Kafka 消息 → Consumer 消费
  → (1) Tika/LiteParse 解析文本
  → (2) 文本切块
  → (3) Embedding 向量化
  → (4) 存入对应 ES 索引
  → (5) 更新 FileUpload 状态
```

### 6.5 路径前缀统一（优先级：中）

**方案 A（推荐）**：前端 axios baseURL 改为 `/api/v1`

**方案 B**：Nginx 反向代理

```nginx
location /api/v1/ {
    proxy_pass http://backend:8081;
}
```
